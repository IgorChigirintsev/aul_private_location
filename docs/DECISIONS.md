## 2026-07-20 — Live Funnel test surfaced three mobile-web bugs; fixed and verified

### D-0067 · A phone-in-browser was a duplicate "PC" on a cramped screen — device idempotency, a 'web-mobile' platform, and responsive dashboard
- **Context:** the first real end-to-end self-host test (a phone on cellular reaching a laptop through Tailscale Funnel — the reachability spike essentially PASSED: live WS survived Funnel) surfaced three UI/correctness bugs the owner spotted: one physical phone showed as **two markers**, both wrongly badged **"PC"**, on a dashboard **not laid out for a phone**.
- **Root cause of the doubling (a general web bug, not phone-specific):** web device creation was not idempotent. The client sent no `device_id` and never persisted the one the server returned, so every re-authentication (cookie expiry / re-login) hit `resolveOrCreateDevice`'s create branch and minted a NEW `devices` row for the SAME persisted identity — and the live map draws one marker per device. Over time any web user would sprout duplicates.
- **Fix, two layers, no migration/sqlc needed:** (1) client persists the returned `device.id` (`keystore.saveDeviceId`) and sends it on the next login so the server reuses that device; (2) server defence-in-depth — when no `device_id` is supplied but a valid identity `pubkey` is, `resolveOrCreateDevice` now scans `ListDevicesForUser` and adopts the row with the matching pubkey before creating (the pubkey is the device's stable identity). Both existing Querier methods, so it works on Postgres and SQLite with no new query. **Verified at the API:** register → login again with the same pubkey and NO device_id returns the SAME device id (no duplicate).
- **The "PC" badge:** a phone in a mobile browser was `platform:'web'`, and `platform === 'web'` drove both the map "PC" badge and the desktop-only "located via Wi-Fi" hint. Fix: `detectPlatform()` (prefers `navigator.userAgentData.mobile`, UA-sniff fallback) registers mobile browsers as **'web-mobile'**; the existing `=== 'web'` checks then correctly exclude phones (badge and hint become desktop-only, like the native app). Server widened to accept it: `validatePlatform`, a Postgres CHECK migration (`00009_web_mobile_platform`), and the SQLite consolidated schema. **Verified:** `register(platform=web-mobile)` → 201; mobile session shows no "PC" tag.
- **Responsive dashboard (all `md:`/pointer-gated, zero desktop change):** the 9-button toolbar wraps into a rounded card on narrow screens instead of overflowing; the MapLibre +/- zoom control is added only for `pointer: fine` (pinch-zoom on touch, and it stops crowding the toolbar); precision-toggle tap targets grow to ~44px on mobile; the People bottom sheet gets a grab-handle; the precision cluster drops to `top-28` on mobile so a centered SOS banner can't overlap it; the Places panel goes full-width on mobile. **Verified by eye** in a 390×844 Playwright screenshot.
- **Verification:** `tsc -b` + `oxlint` + `vitest` (122 + a new `detectPlatform` test) green; **full desktop e2e 16/16** against a fresh SQLite server built with the fixes (no regression); the API dedup + web-mobile-accept probes above; and the mobile-viewport run (one marker, no "PC" tag, no zoom control). Built directly, reusing an earlier sound diagnosis and plan.
- **Still owed (owner-flagged):** the fix stops NEW duplicates but **existing duplicate device rows persist** (on the live GCP DB and any already-affected launcher) — and because each device's newest ping never prunes, a stale duplicate marker lingers. Needs a one-time cleanup (delete duplicate `(user_id, pubkey)` rows keeping the most-recently-seen) and ideally a device-removal UI; a hard `UNIQUE(user_id, pubkey)` index is deliberately NOT added yet because it would fail against the live rows until that cleanup runs.

## 2026-07-20 — Self-host Milestone 3 begins: a pure-Go launcher core, tray/signing split off

### D-0066 · The launcher is split — a pure-Go headless CORE now, the native tray + signing later
- **Context:** Milestones 1&2 gave a working single pure-Go SQLite server binary (D-0065, runtime-verified). Milestone 3 is "run it, one button." The design's tray UI (menu-bar icon) inevitably pulls a native GUI toolkit → **CGO on macOS/Linux**, which would poison the very no-cgo property that made `modernc.org/sqlite` worth the dual-dialect tax. And the tray's remaining work (per-OS signing/notarization, EV-cert legal-entity question) needs the **owner** anyway.
- **Decision:** split the launcher. Ship the **headless CORE now** as pure Go (`server/internal/launcher/` + `server/cmd/aul-launcher/`, same module, `CGO_ENABLED=0`), and defer the native tray + signing to Phase 2 where the owner's certs are the gate. The tray becomes a thin CGO shell over this verified core, not a rewrite. Built, then **independently re-verified** (green tests ≠ works).
- **What the core does:** provisions a data dir (`os.UserConfigDir()/aul`, 0700) and a **persistent session pepper** — generated once from `crypto/rand`, written atomically 0600, **never rotated** (rotating re-hashes every session token → silently logs the whole circle out; a documented landmine, so `provisionPepper` reuses the file verbatim); composes the child's env (SQLite `DATABASE_URL`, `RUN_MIGRATIONS`, and — deliberately — *never* `AUL_DB_BACKEND`, which would force Postgres and break the `sqlite:` inference); **resolves the public origin** in priority `--origin` > detected Tailscale Funnel > `http://localhost:PORT` fallback, with the one coupling that matters (`https`→production+secure-cookies, `http`→development+insecure — the server hard-fails secure cookies over http); **spawns and supervises exactly one** server child (capped-backoff restart, graceful `SIGINT`→grace→`Kill`, no restart on our own shutdown, single-writer via an `O_CREATE|O_EXCL` pidfile lock with liveness-based stale reclaim); polls `/readyz`; and **opens the dashboard at the resolved origin, never at localhost when a real front exists** (baking localhost into an invite is THE critical launcher bug, so it lives in one place).
- **Security hardening caught in review:** the child now binds **`127.0.0.1:PORT`, not `:PORT`**. The public front (Funnel / a Direct-mode proxy) is same-box and reaches the server over loopback while terminating TLS; binding all interfaces would additionally expose the **plaintext** http server — credentials, session cookies, the social graph — to the whole LAN, straight past that TLS. `localhost` origins still resolve (happy-eyeballs) and the `/readyz` probe uses `127.0.0.1`, so nothing legitimate loses reach.
- **Verified by me, at runtime, not on report:** `CGO_ENABLED=0` builds both binaries + whole module; 17 launcher unit tests green; a launcher started against a fresh data dir provisioned pepper(0600)+db+lock+log, brought the server up (`/readyz` 200 on **both** `127.0.0.1` and `localhost`), and `ss` confirmed `LISTEN 127.0.0.1` only; **the real E2EE e2e suite (e2ee-proof + acceptance + share = 11/11) passed against the launcher-spawned SQLite server** — the composed env serves correct end-to-end encryption; pepper stayed byte-identical (sha256) across a stop+restart; a second launcher on the same data dir refused; and `SIGINT` to the launcher stopped the server child with no orphan on the port.
- **Still owner-gated (unchanged from D-0065):** the native tray + per-OS signing/notarization; the Phase-0 Tailscale reachability spike (does the WS hub survive Funnel to a phone on cellular — runbook in [SELFHOST_PHASE0_REACHABILITY.md](SELFHOST_PHASE0_REACHABILITY.md)); and the FCM-in-self-host / offline-signal / disk-failure-backup decisions.

## 2026-07-20 — Self-host-on-PC: design chosen, and the SQLite backend port begins

### D-0065 · Self-host on your PC = pure-Go SQLite + Tailscale Funnel + a signed native binary; the cloud stays on Postgres (two dialects, forever)
- **Context:** the owner wants "run your own Aul server on your PC, one button, any OS." The three hard forks were evaluated (full writeup in [SELF_HOST_DESIGN.md](SELF_HOST_DESIGN.md)).
- **Decisions:** (1) **DB** — embed **`modernc.org/sqlite`** (pure-Go, no cgo) as a second backend so the self-host build is a single file-based binary; the cloud keeps **Postgres** (it needs partitioning/concurrency/scale). The permanent price, accepted knowingly: **two SQL dialects maintained forever**, with dual-backend CI as a release gate — because the dialect-hostile queries are exactly the privacy paths (who-sees-whose-location, who-is-muted), so a mirror that drifts becomes a self-host-only location leak. (2) **Reachability** — the owner chose **Tailscale Funnel by default + a "Direct mode" (port-forward) toggle**; **never** Cloudflare (it terminates your TLS → sees emails/passwords/graph) and **never** a relay through our own server (re-centralizes us). Funnel's relay is metadata-only and on the *user's own* account, and TLS terminates on the home box. (3) **Packaging** — a single signed native binary per OS with a tray UI.
- **Owned honestly (from the critique), to resolve before GA:** "one button" is really "one button to run + a short guided setup" (Tailscale needs an account, a privileged daemon, and a console Funnel toggle no launcher can automate); **FCM is an unlisted third party** — Android push can't be P2P, so self-host must pick route-through-our-Firebase (re-centralizes metadata) / bring-your-own-Firebase (not one-button) / **Android-push-off** (leaning here, disclosed); the **"server offline" signal must be client-inferred** (an off server can't announce itself) and must ship before any non-dogfood exposure, or a stale dot looks fresh on a safety app; and hardware replacement / disk failure need an origin-rebind + off-box backup story.
- **Milestone 1 landed (verified):** a parallel SQLite migration set (`db/migrations_sqlite/`, consolidated end-state) materialises the full schema; `NULLS NOT DISTINCT` → **two partial unique indexes**, independently adversarially checked to match Postgres exactly (M1/M2 distinct members don't false-collide, whole-circle NULL coexists, no dups); a standalone open+migrate helper with `foreign_keys=ON`/WAL/busy_timeout PRAGMAs; config backend detection defaulting to Postgres. `CGO_ENABLED=0` build+tests green (pure-Go proven); Postgres path byte-identical.
- **Milestone 2 landed (verified) — the server actually runs on SQLite.** A **hand-written `database/sql` store** implementing the existing `store.Querier` (no second sqlc codegen — sqlc's SQLite engine can't emit `uuid.UUID`/`time.Time` round-tripping types; a `var _ Querier` guard forces every future Postgres query to gain its SQLite mirror). The 6 dialect-hostile queries were rewritten and **re-proven equivalent, not just translated**: `= ANY($1::uuid[])`→expanded `IN`; `DISTINCT ON`→`ROW_NUMBER()…=1`; `interval`/`make_interval`/`now()` predicates→Go-computed cutoff bounds; `'epoch'::timestamptz`/`::casts`; `InsertMute` upsert→bare `ON CONFLICT` on the two partial indexes; partition machinery→DELETE-by-timestamp. A uuid TEXT Scanner/Valuer wrapper + a **fixed-width 3-digit-ms `…Z` time format** (load-bearing — SQLite sorts TEXT lexicographically, so `RFC3339Nano`'s trimmed zeros would mis-order time windows). Verified by me on **both** backends: full `-tags=integration` suite green under `AUL_TEST_BACKEND=sqlite` AND against throwaway Postgres; the 3 privacy-critical equivalence tests (live-map latest-fix, mute-set count, mute dedup+FK-cascade) run on both and agree; FK cascade proven live on a pooled conn; and a `CGO_ENABLED=0` binary booted against a real `sqlite:` file drove register→circle→ping→pings/latest. The `publish-version` tool was made backend-aware too. The single pure-Go SQLite binary now works end-to-end; Milestone 3 (the native launcher + tray + Tailscale bundling, cross-OS signing) is next and needs the owner (Tailscale account, signing certs) + the unresolved FCM/offline-signal decisions.

### D-0063 · A share-revoke test was a time bomb — `ShareKeyStore.add` stamped `addedAt` off the wall clock
- **Symptom:** `background_shares_test` ("a revoke from another device stops the feed within the TTL") passed on 2026-07-16 and began failing deterministically days later — a second position ping went out after a revoke. Nothing in the share code had changed since.
- **Root cause:** the test pins the feeder to a fixed clock (`now = 2026-07-16 12:00`), but `ShareKeyStore.add` stamped `addedAt` with the real `DateTime.now()`. `sync` keeps a session whose `addedAt` is after a refresh's start time (created mid-fetch ⇒ a stale empty response must not prune it). Once **wall time moved past the test's fixed date**, every seeded session looked "created after the fetch" to that guard, so a revoke stopped pruning and the feed continued. Production is unaffected — there both `addedAt` and the refresh's `startedAt` are real wall time, so the guard compares one clock. The test compared two.
- **Fix:** `add(..., {DateTime? now})` — injectable, defaulting to the wall clock; the test's `seed` passes its fixed `now`. Deterministic now, and mutation-checked (disabling the TTL refresh makes the revoke test fail, proving it guards the behaviour rather than the date). The lesson: an injectable clock has to reach *every* timestamp the logic compares, or a test that looks time-controlled is still racing the wall.

### D-0064 · The public server runs free on GCP e2-micro — after Oracle's free ARM proved unobtainable
- **Context:** the owner wanted a public server holding ~1000 users, free. Oracle Always Free ARM (24 GB, the only free box with real headroom) was the target, but Frankfurt had **zero** free-ARM capacity across ~44 hours and ~1000 API launch attempts (all sizes, all ADs). An OCI-CLI auto-retry keeps running in the background to grab it the moment capacity frees, for a later migration.
- **Reason:** to have a working server *today*, deployed to **GCP e2-micro** (Always Free: 1 e2-micro in us-west1/-central1/-east1, 30 GB standard disk, non-preemptible). It is x86/1 GB — tight for 1000 but fine for launch, and the server is a light ciphertext relay. Live at **https://<ip>.nip.io** with real Let's Encrypt TLS via Caddy (nip.io wildcard-resolves the IP → no domain to register), the official `postgres:16` (D-0061 made the stack ARM/x86 multi-arch and PostGIS-free), a 2 GB swap file (1 GB RAM needs it), and a daily `pg_dump` cron. Registration, web UI, and HTTPS verified end-to-end from outside.
- **Known, flagged:** the GCP external IP is **ephemeral** — a stop/start changes it and breaks the nip.io hostname; reserving a static IP is the pending fix. Migration to Oracle ARM, if the retry ever lands it, is backup→restore.
- **Push enabled 2026-07-20:** VAPID (web) generated on-box; FCM (app) via the service-account key mounted `./fcm:/fcm:ro`, verified with a live `/notify` reaching Google (a bogus token ⇒ `400`, not an auth `401/403`). Two ops gotchas cost time and are worth remembering: a bind-mount dir populated *after* the container was created needs `--force-recreate`, not a restart; and `docker compose exec/run` **hangs inside a `bash -s` heredoc** because it reads the heredoc as its stdin — run it as a script file or `</dev/null` it.

## 2026-07-16 — The database drops PostGIS, so the stack runs on free ARM

### D-0061 · Official `postgres:16`, no PostGIS — for trust and for ARM
- **Context:** planning a free public host (Oracle Cloud Always Free). Its generous tier is **ARM** (Ampere A1, 4 vCPU / 24 GB), but `postgis/postgis:16-3.4` publishes **only** an amd64 manifest — it will not run there. The server image itself is fine on ARM (CGO off, distroless static, built on-host).
- **The finding that made it easy:** PostGIS was `CREATE EXTENSION`-ed in migration 00001 "for a future trusted-server geofencing mode" and **never used** — no `geometry`/`geography` column anywhere, no `ST_*` call in any query. It was dead weight that happened to also block the free host.
- **Alternatives:** a community multi-arch PostGIS image (`imresamu/postgis`, which does ship arm64) — rejected: adding a third-party image maintainer to the trust base of an E2EE product, to keep an extension nothing calls, is exactly the wrong trade; the owner chose against it. Run on Oracle's AMD free tier instead (1 GB / 1 core) — rejected: 24× less RAM to dodge a one-line change.
- **Reason:** dropping PostGIS moves the DB to the **official** `postgres:16` image — more trustworthy than any third party, and multi-arch (amd64 + arm64), so the same compose runs on the free ARM box and on any x86 host. `pgcrypto` (`gen_random_uuid`) and `citext` (case-insensitive email) stay — both ship in the official image's contrib. If trusted-server mode is ever built, it reintroduces PostGIS in its own migration then, and only self-hosters who opt into it need a PostGIS-capable image.
- **Verified:** migration 00001 edited to stop creating the extension; the image swapped in all four places it was pinned (deploy compose, CI, both Makefile targets). A **fresh official `postgres:16`** (image id confirmed, not postgis) ran every migration from scratch and the full `-tags=integration ./internal/...` suite passed — proof the schema never depended on PostGIS, not just an assertion that it shouldn't. Editing an already-applied migration is safe here: goose keys by version, so existing databases don't re-run it, and the change only governs fresh deploys — which is the entire point.

## 2026-07-16 — A dead channel, three features that never ran, and a false arrival

### D-0059 · `_onForegroundFix` had never executed. Live share had never sent a position. The location isolate now owns every fix.
- **Context:** an audit of "arrival alerts only fire while the app is open" found the premise too generous. `LocationControl.setForegroundLocationHandler` registered for method `onLocation` on the **`app.aul/control`** channel. Nothing has ever sent it: Android's `LocationService.kt:316` forwards to **`app.aul/bg`**, and iOS's `LocationBridge.swift:77` does too. `MainActivity` only *receives* on the control channel. So `AppController._onForegroundFix` never ran on either platform — and it was the only caller of three things.
- **What was dead, in order of severity:** (1) **Live share never sent a single position.** A user creates a 60-minute link, hands it over, and the viewer waits forever. Worse, `_startShareOnlyLocation()` faithfully *started the native service* for that share — so the phone burned battery streaming fixes into a handler registered on a channel nobody sends to. (2) **Geofence crossings never fired**, so arrival alerts never worked and the mutes/relay machinery (D-0053/D-0055) governed a fan-out nothing triggered. (3) The low-battery reminder never fired. A second corpse sat one layer down: `ShareController._pingTimer` re-sent `_lastFix` on a timer, and `_lastFix` was permanently null.
- **Decision: the location isolate owns evaluation, exclusively, and the foreground path is deleted** — not deprecated, deleted, along with the control-channel handler and all six call sites. The isolate is the only Dart with a fix stream and it runs whenever tracking is on, so evaluating in the UI adds no coverage — only duplicate notifications and duplicate `/notify` fan-outs. "The other one stops by convention" is not a mechanism; the other one no longer exists.
- **Three hard parts, and what they cost:**
  - **Durable crossing state** (`tracking/geofence_state.dart`, kept in the KeyVault beside K_c). `GeofenceEngine._inside` was memory-only, and Android restarts a foreground service freely: a fresh engine starts empty, so the first fix at home is an `enter` — **every service restart would re-announce "arrived at Home" to the whole circle**. The keystore over the drift DB: writes are rare (a crossing, a few a day) and never on the hot path, it keeps the `queued_pings` posture that only ciphertext rests on disk, and `wipe()` gives sign-out cleanup free. *Trade:* no transactions — safe only because there is exactly one writer, which the decision above guarantees.
  - **Places in the isolate**: the foreground seeds a cache of the server's **own ciphertext** (a test asserts the at-rest bytes hold no name and no coordinate), decrypted in-memory with K_c; the isolate refreshes on a 15-min TTL. Fetch-per-wake is ~1400 requests/day to re-learn a monthly-changing list and goes blind offline; a pure foreground cache goes stale forever. *Trade:* a place added elsewhere is live within 15 minutes, not instantly.
  - **Share sessions live in the KeyVault** (`share_sessions`), which also **fixes a real weakness**: `ShareKeyStore` kept K_share in **cleartext SharedPreferences**. It is key material of exactly the kind the vault exists for. The isolate re-reads it every fix (a share created 30 s ago must be fed now — local read, no radio), plus a 30 s server refresh to catch a revoke from another device; the deadline is enforced per fix against the cached `expiresAt`, independent of that refresh. Share fixes go **RAW** — never coarsened by the circle's mode, never gated on the circle's targets: a link is its own opt-in with its own key. Share pings are deliberately not queued like circle pings: a position minutes late is worthless to a watcher and the next is seconds away.
- **The two gates survive intact** (D-0055): the arrival opt-in governs whether *you* are buzzed; the operator kill-switch alone governs whether the circle is told. The kill-switch was memory-only in `RetentionController`, so the isolate could not read it; the foreground now mirrors it to `SharedPreferences`, **defaulting false** to preserve the anti-stalking invariant. *Cost:* turning it off server-side stops a device at its next foreground launch, not instantly.
- **iOS is worse than "not built" — it is wired to nothing.** `LocationBridge.swift` puts `app.aul/bg` on the **main** messenger, but the only Dart handler for that channel is `BackgroundReporter`, constructed inside `aulLocationServiceMain` — an entrypoint **only Android's `LocationService` ever spawns** (`main.dart` merely holds a reference so it survives tree-shaking). On iOS no fix reaches Dart at all: not shares, not pings, not geofences. Today's work is correct there the moment that engine is wired; it does not wire it. Recorded so nobody re-discovers it the hard way.

### D-0060 · A first look at a (device, place) pair is **state, not news** — the web was relaying arrivals at places nobody had left
- **Context:** `GeofenceFeed` guarded announcements with a `seeded` boolean: "the first pass with real data is state, not news". Two bugs hid in it. **`setEvents` sat outside the gate**, so every reload drew phantom "arrived at Home" rows for everyone already home. And the latch was **one flag for the whole feed** while the condition is **per pair**: `FRESH_MS` filters out positions older than 15 minutes, so leave a laptop at home overnight and reload — its stale fix is filtered away, other members' fresh phones close the latch, then the laptop reports and the tracker sees it for the first time → `enter` → **a real Web Push to the entire circle saying you arrived somewhere you never left**.
- **Alternatives:** make the latch per-device in the component (still wrong: a place created around a stationary device announces an arrival too); drop the relay for late-appearing devices (silently loses real arrivals).
- **Reason:** The rule belongs to the tracker, because only it knows which pairs it has evaluated. `GeofenceTracker` now seeds per `(device, place)`: the first look records state and returns nothing, so the caller's latch — and the class of bug it invited — is gone, and the feed, the local alert and the relay all act on one set that cannot disagree. Pruning changed with it: state is dropped when a **place** is deleted (a fence that no longer exists), but **not** when a device merely goes quiet. Forgetting a quiet device makes its return either a silent first look (a real arrival lost) or an arrival at a place it never left; remembering means a device back where it was says nothing, and one back elsewhere exits — which is what actually happened.

The fourth ("goes quiet, comes back to the same place") passed *with or without* the fix — both versions are silent there, one because it remembers and one because it re-seeds. - **Found while editing:** `data/geofence.ts` was **not a text file**. Two literal NUL bytes were typed into the source as the composite-key separator (`` `${deviceId}\0${placeId}` ``). It worked — a NUL cannot occur in a UUID — but `file` called it `data`, string-matching tools could not edit it, and the separator was invisible in review: fix one of the two occurrences and the keys silently stop matching. ## 2026-07-15 — Background push lands on the phone (FCM)

### D-0057 · FCM is a second push channel, **data-only**, and the Firebase project belongs to the operator — not to the repo
- **Context:** D-0055 closed app parity except one hole that was not code: a phone could *send* arrival alerts (web clients got them via Web Push, D-0049) but could not *receive* one while backgrounded. Waking a closed Android app is not something an app may do by itself — the OS insists on its own relay. The owner created the Firebase project; this decision records what was built on it.
- **Alternatives:** FCM `notification` messages (**rejected — it breaks E2EE**, see below); the Firebase Admin SDK for Go (rejected: it drags in the whole auth/firestore/storage/gRPC stack, versus **2 modules** for `x/oauth2/google` + one POST — in a codebase whose security posture *is* the product, dependency surface is a feature); a persistent socket instead of a relay (battery, and the OS fights it); UnifiedPush (not built — see the honest limit below).
- **The E2EE rule:** messages are **data-only**. An FCM `notification` message is rendered by the Android system itself, which would force the plaintext through Google — the exact thing Aul exists to prevent. So the payload is the same opaque blob the sender sealed under K_c (AD `aul-notify:v1`), and the phone decrypts it and raises a *local* notification. This is not a convention to be remembered: `internal/fcm`'s message struct has **no notification field**, so no caller can add one by accident. Three tests at three layers assert it structurally and textually; the guard was verified by *actually* adding a `notification: {"body": "Anna arrived at Home"}` to the sender and watching all three fail with `E2EE VIOLATION` (the first mutation attempt was a silent no-op that "passed" — proof that a green mutation test is worth exactly nothing until you confirm the mutation applied).
- **Schema:** `push_subscriptions` was Web-Push-shaped (`endpoint` URL + `p256dh` + `auth`); an FCM registration token has none of that. A `kind` discriminator (`webpush|fcm`) carries the token in `endpoint`, because it plays the identical role — the opaque per-device address — which keeps **one** unique key, one prune path and one unsubscribe query for both channels. `p256dh`/`auth` drop NOT NULL, and a per-kind CHECK (`webpush ⇒ keys present, fcm ⇒ keys NULL`) forbids a half-filled undeliverable row at the database, not merely in the handler. `Down` must delete fcm rows — they have no representation in the old schema.
- **`INVALID_ARGUMENT` deliberately does not prune.** 404/`NOT_FOUND`/`UNREGISTERED` mean a dead token and delete the row (sharing Web Push's prune path). But `INVALID_ARGUMENT` is also what Google returns for *our own* malformed request — pruning on it would wipe **every subscription in the fan-out** on a bad deploy. Verified live: a bogus token yields `{"sent":0,"failed":1}` and the row survives.
- **The Firebase project is per-operator, and `google-services.json` is untracked.** It is *not* a secret (it ships inside the APK; anyone can extract it), but committing it would bind every fork and self-host to **one** project — a forked APK would register tokens against the original owner's Firebase. So it is gitignored, and the Gradle plugin is applied **only when the file exists**: a contributor cloning without a Firebase project still gets a building APK with push inert. The *server's* service-account key is a genuine secret and is passed as a **path** (`FCM_SERVICE_ACCOUNT_FILE` + a read-only mount), never as an env *value* — an env var would sit in `docker inspect` output forever. Unset ⇒ FCM disabled and the server boots, mirroring VAPID.
- **The background isolate is never handed K_c.** It re-reads it from the same OS keystore the UI isolate writes to (`KeyVault(FlutterSecretStore())`), exactly as the existing headless location isolate does — so the circle key never crosses an isolate boundary and is never serialized into a payload. Since the push carries **no circle id** (that metadata is precisely what Aul declines to leak), decryption tries every key the device holds — rotation-safe, mirroring the web service worker. **Undecryptable ⇒ nothing is shown.** The web's generic "Activity in your circle" fallback was deliberately *not* copied: it exists only because Chrome's `userVisibleOnly:true` extorts *some* visible notification, and Android imposes no such bargain on data-only messages, so silence is strictly better than a content-free buzz.
- **The honest costs**, both now in THREAT_MODEL §4: (1) **Google sees the traffic pattern** — that a push reached a device, when, and its size — on top of what the server already knows. It reads nothing. This is the price of waking a closed app, and it is opt-out: leave VAPID and FCM unset and no third party is in the picture (at the cost of no background alerts). (2) **FCM requires Google Play Services**, so a de-Googled phone (GrapheneOS, /e/OS) — plausibly a chunk of this product's audience — gets no background push at all. **UnifiedPush** (self-hosted ntfy as the relay) is the standing answer and is *not built*; it is recorded here as a known gap rather than discovered later.
- **Verified:** the service account mints a real Google access token; FCM v1 accepts the credential and the project, rejecting only a deliberately-bogus registration token (`400 INVALID_ARGUMENT` — a 401/403 would have meant a broken key or a disabled API); the live server boots `push:true fcm:true`, stores a `kind=fcm` row, and a real `/notify` performs a genuine round-trip to Google. Uncached `go test -tags=integration` green; `flutter build apk` green (analyze/test do not compile native — D-0043's lesson). **Not verified: delivery to a real handset** — no device here, so the first real push is unproven. Also unbuilt: tap-to-open-the-circle (the app has no navigator key or route table; it would mean changing the shared notification interface, so it was left whole rather than half-done) and iOS (needs APNs + a `GoogleService-Info.plist`).

## 2026-07-15 — E2EE Web Push, movement stats, member removal, code-split

### D-0049 · Web Push carries a **sealed** blob the service worker decrypts — the server relays ciphertext it cannot read
- **Alternatives:** push the plaintext "Ann arrived at Home" (the server would see it — unacceptable); push a contentless "open the app" ping (safe but useless); compute arrivals server-side (impossible — the server has no coordinates, D-0035).
- **Reason:** Arrival alerts only worked while a dashboard tab was open, because geofences are computed client-side. Now the **arriving** client seals `{t,place,who,ts}` under K_c (`sealFramed`, AD `aul-notify:v1`) and POSTs it to `POST /v1/circles/:id/notify`; the server fans it out over Web Push (VAPID, RFC 8291/8292) to the circle's *other* members' subscriptions, pruning 404/410 rows. The payload is therefore encrypted **twice** — E2EE under K_c, then again by Web Push to the subscription — and the server (and the push service) only ever relay the opaque blob. The **service worker** decrypts it with libsodium + the keys in IndexedDB and shows the real text; when nothing opens it (other circle / no key) it shows a deliberately contentless fallback rather than leaking. Config: `VAPID_PUBLIC_KEY`/`VAPID_PRIVATE_KEY`/`VAPID_SUBJECT`, all optional (unset ⇒ push disabled, server still boots); `aul vapid-keys` generates them; `/v1/server-info` advertises the public key so clients can subscribe. ## 2026-07-14 — Phase 5: Places, geofences, history, SOS, precision

### D-0037 · Phase-5 adversarial-review fixes (2 medium correctness/safety + hardening)
- **Context:** An adversarial security review (4 dimensions, reviewer → independent verifier) found **no critical/high**; it confirmed the server access-control and retention paths **secure** (no cross-circle IDOR, correct optimistic concurrency, no active-SOS/live-place deletion) and surfaced 2 medium + several low issues, all fixed:
- **[MEDIUM] Stale SOS poll clobbered a live alert.** The 60 s `listSos` poll did `setAll`, so an in-flight poll returning before a realtime-added SOS would drop it — hiding an emergency for up to a minute. - **[MEDIUM] Viewer-side geofence presence went permanently stale.** A device that stopped reporting kept its last position, so it appeared "inside" forever. Fixed: `GeofenceFeed` ignores positions older than 15 min and re-evaluates on a 30 s timer, so presence ages out.

- **[LOW] History truncation dropped the *newest* pings** (`ORDER BY captured_at ASC LIMIT`). Now ordered **newest-first with the LIMIT** and reversed in the handler, so a truncated window keeps the most recent track.
- **[LOW] Domain separation** (see D-0034): places/SOS now seal with associated data (`aul-place:v1` / `aul-sos:v1`) so a ciphertext can't be replayed across field types.

### D-0034 · Places & SOS payloads are single **framed, padded** blobs sealed with K_c (reuse the circle-name shape, add padding)
- **Alternatives:** the ping two-field `{nonce, ciphertext}` shape; store name/coords in separate columns (breaks E2EE); no padding.
- **Reason:** `places_enc` and `sos_events` each have a **single** opaque `ciphertext bytea` column (no separate nonce), so a place/SOS must be self-framed: `sealFramed(pad(utf8(json), 256), K_c)` → `nonce(24B)||ciphertext`, base64-std, exactly the circle-name layout — **not** the ping's two-field split. The client JSON uses terse keys (place: `{n,lat,lng,rad}` metres; SOS: `{lat,lng,msg,ts}`). Unlike the circle name (which is unpadded — a small length leak, accepted; see THREAT_MODEL), places/SOS are **padded to a 256-byte block before sealing** so a place name (`"Mom's house"` vs `"Gym"`) or a short SOS message doesn't leak its length via ciphertext size. This hides length *within* the 256-byte block; an oversized payload buckets up to 512 (a coarse "large vs small" bit), so the UI caps the place name (80 chars) and SOS message (160) to keep normal payloads in the single-block bucket — the residual 256-byte-granularity is documented in THREAT_MODEL, not claimed away. Places and SOS also seal with **domain-separation associated data** (`aul-place:v1` / `aul-sos:v1`) so a ciphertext of one type cannot be opened/replayed as another (added after the Phase-5 review; pinned in the `place_framed` vector, which also checks a wrong-AD open fails). Blobs are opened across the **whole circle keyring** (rotation-safe, like pings). This needed **no new AEAD primitive** (XChaCha20-Poly1305 is already cross-tested/pinned); the only crypto additions are Dart `sealFramed/openFramed` (web already had them) and a cross-language **framed+padding vector** so the byte layout is pinned, not just round-tripped per language.

Enter/exit is therefore computed by clients that hold K_c: the **web dashboard** computes live geofence state (who is inside which place) from decrypted live positions vs synced places, with enter/exit hysteresis, and shows arrive/depart in-session; the **reporter app** has a pure-Dart `geofence_engine` (unit-tested, hysteresis/debounce) over its own fix stream. ## 2026-07-14 — Phase 4: E2EE hardening across the stack

### D-0033 · Map container positioning is set inline, not via Tailwind — `maplibre-gl.css` is unlayered and beats `@layer utilities`
- **Alternatives:** keep the `absolute inset-0` Tailwind className; import `maplibre-gl.css` into a cascade layer so Tailwind utilities win; give the container `h-full w-full` instead of absolute positioning.
- **Reason:** A second, independent cause of the blank map (found after the CSP fix, via a live DOM diagnostic showing the map container was `848×0`): Tailwind v4 emits its utilities inside `@layer utilities`, but `maplibre-gl.css` ships `.maplibregl-map { position: relative }` **unlayered** — and unlayered CSS always beats layered CSS regardless of source order/specificity. (Related: D‑0025 map rendering, D‑0032 tiles CSP.)

### D-0032 · `TILES_ORIGIN` defaults to OpenFreeMap so the CSP allows the map out of the box (blank-map fix)
- **Alternatives:** leave `TILES_ORIGIN` empty by default (require every operator to set it); ship a laxer CSP; hardcode the origin in the middleware.
- **Reason:** The web client points MapLibre at `https://tiles.openfreemap.org` by default (D‑0018/D‑0025), but the server's `TILES_ORIGIN` had **no default** — so a server started without it emitted `connect-src 'self'` (and `img-src` without the tiles host), and the browser silently blocked the style JSON, vector tiles, glyphs, and sprites → **the map rendered blank** (only the DOM marker overlay showed, which is why the Playwright acceptance test — checking `.aul-marker` — never caught it). Guarded by `TestLoad_TilesOriginDefaultsToOpenFreeMap` (config), `TestCSP_AllowsMapTiles` (middleware asserts the origin is in `connect-src` **and** `img-src`, plus `worker-src blob:`/`wasm-unsafe-eval`), and a strengthened Playwright acceptance test that now asserts the basemap tiles actually load (≥1 tile OK, none blocked), not just that the marker overlay appears.

Also fixed the **PWA service worker** that made the bug sticky: it precached `index.html` and served it via Workbox's precache navigation fallback, so the cached document pinned the *old* CSP header in the SW cache — a plain reload kept replaying the stale CSP even after the server was corrected. The app shell is no longer precached; navigations now use a **NetworkFirst** runtime route (`navigateFallback: undefined` + a `request.mode === 'navigate'` handler), so the document's headers (CSP included) are always fresh online with the cached copy only as an offline fallback. Content‑hashed static assets are still precached. (A browser that already cached the old page once needs a one‑time SW unregister / "Clear site data" to drop the stale cache; new loads and future header changes are clean.)

### D-0031 · Key envelopes are distributed at the circle's **current epoch**; rotation bumps the epoch *before* distributing (Phase‑4 security‑review fix)
- **Alternatives:** hardcode envelope epoch to `1` (original); have clients send an explicit epoch they compute locally.
- **Reason:** The Phase‑4 adversarial review found a **rotation‑correctness bug** (MEDIUM): the server hardcoded `key_epoch = 1` for any envelope submitted with `key_epoch ≤ 0`, while both clients send `0` and *document* "server clamps to the current epoch". Because envelopes upsert on `(circle_id, recipient_device_id, key_epoch)` (migration `00004`), every rotation overwrote the **same** epoch‑1 row — so a device offline across **two** rotations only ever received the newest key, and pings sealed under the intermediate key became permanently unreadable on that device (silent history loss; the keyring in D‑0029 was designed to prevent exactly this). Fix: the server now clamps `≤0` to the circle's real `key_epoch` (`GetCircle`), giving one distinct envelope row per rotation so an offline device catches up on *every* intermediate key. The review also caught an off‑by‑one in the naïve fix (distribution ran *before* the epoch bump); so both clients' `rotateKey` now **bump the epoch first, then distribute**, landing the new key at the new epoch. Guarded by `TestAPI_KeyEnvelopes_PerEpochDistribution` (two rotations ⇒ three distinct pending envelopes for an offline device; re‑delivery at an unchanged epoch upserts in place). Confidentiality/integrity were never affected — this is an availability/correctness fix.

### D-0028 · `crypto_box_seal` via `nacl/box.SealAnonymous` (Go) — libsodium‑compatible, cross‑tested
- **Alternatives:** a Go libsodium binding; hand‑rolling the sealed‑box construction.
- **Reason:** `golang.org/x/crypto/nacl/box`'s anonymous sealed box is byte‑compatible with libsodium's `crypto_box_seal` (ephemeral key + BLAKE2b nonce). Confirmed empirically: **Dart and JS both open a box Go sealed** (a committed vector in `/vectors/crypto-vectors.json`). The vector is made reproducible by sealing with a fixed ephemeral seed (box_seal is otherwise nondeterministic) so the auto‑compared vectors stay stable. This gives three‑way (Go↔Dart↔JS) interop for the key‑envelope path — the Phase‑4 criterion.

### D-0029 · Circle key stored as a per‑circle **keyring** (retain old keys); v1 has no forward secrecy
- **Alternatives:** overwrite K_c on rotation; store per‑epoch keys keyed by epoch.
- **Reason:** Pings don't carry which epoch sealed them, so after a rotation a client must be able to try the old key on old history. The web keystore holds a keyring per circle (newest last); decryption tries each key (rotation‑safe), and the newest is used for new invite fragments. This honors "old history stays readable" (spec §3). It also makes explicit that **v1 has no forward secrecy** — a member who kept an old key can still read old history; MLS group keying is the roadmap (documented in THREAT_MODEL §6). (The reporter app keeps a single current key since its job is to *seal*, not view history.)

### D-0030 · Key distribution: invite fragment for the first join, sealed envelopes for multi‑device + rotation
- **Alternatives:** always use envelopes; always use the fragment.
- **Reason:** A first join is simplest via the invite‑link **fragment** (K_c never hits the server). But a *second* device of the same user (no fragment) and **rotation** (redistribute a new K_c to all remaining devices) need `crypto_box_seal` **envelopes**: the sender seals K_c to each member device's X25519 identity public key (fetched from the new `GET /v1/circles/:id/devices`), posts them, and each device opens its own with its private key. The server validates recipients are member devices and relays boxes it cannot open. Clients call `openPendingEnvelopes()` on startup/login to pick up keys. Rotation bumps the server key‑epoch and is offered to owners (and appropriate on member removal). Both the web and Dart `KeyManager`s were validated end‑to‑end against the live server.

## 2026-07-14 — Phase 3: Web dashboard

### D-0024 · Web authenticates with httpOnly cookies (no tokens in JS)
- **Alternatives:** store access/refresh tokens in `localStorage`/memory and send `Authorization` headers.
- **Reason:** The server already issues `httpOnly`+`Secure`+`SameSite=Lax` cookies on login. The dashboard uses them (`fetch(..., {credentials:'include'})`) and keeps **nothing** sensitive in JS, so XSS can't read the session. The browser also sends the cookie on the same‑origin WebSocket automatically. On 401 the client transparently POSTs `/v1/auth/refresh` once (cookie‑based) and retries. This is more XSS‑resistant than JS‑held tokens, at the cost of relying on SameSite for CSRF (adequate for a Lax same‑origin API; state‑changing routes are POST/PUT/DELETE).

### D-0025 · Map: MapLibre GL + OpenFreeMap "positron"; markers as DOM + rAF interpolation
- **Alternatives:** a full custom warm MapLibre style JSON; canvas symbol layers for markers; Google Maps SDK (banned).
- **Reason:** OpenFreeMap's hosted **positron** style (no API key, self‑hostable, one origin for tiles+glyphs+sprites) is a muted light basemap that fits the warm palette; a bespoke style is polish (roadmap). Member markers are **DOM elements** (`maplibregl.Marker`) so the avatar + battery‑ring (conic‑gradient) + accuracy halo + fresh‑update pulse are plain CSS, and moves are animated by interpolating `setLngLat` over ~1 s via `requestAnimationFrame` (honoring `prefers-reduced-motion`). This is what makes a watcher "see the reporter move".

### D-0026 · React 19 (Vite scaffold default) — satisfies the spec's "React 18"
- **Alternatives:** pin React 18.
- **Reason:** The current Vite template ships React 19, and TanStack Query / Zustand / MapLibre / Framer Motion all support it. "React 18" is a floor; 19 is compatible and current (same reasoning as Go 1.25, D‑0014). No feature depends on 19‑only APIs.

### D-0027 · Web served by the Go binary via `embed.FS`; dev via a Vite proxy
- **Alternatives:** deploy the web as a separate static site/CDN.
- **Reason:** Spec mandate (one binary serves the web). `make web` builds the bundle into `server/cmd/aul/webdist` (git‑ignored build output; a placeholder ships in the repo so `go build` works standalone). In dev, Vite serves on :5173 and proxies `/v1` + the WebSocket to the Go server. Because the server's WebSocket enforces an Origin allow‑list equal to `PUBLIC_ORIGIN`, **dev must run the server with `PUBLIC_ORIGIN=http://localhost:5173`** (or serve the embedded build same‑origin). The strict CSP needed `'wasm-unsafe-eval'` in `script-src` for libsodium's WASM; validated by running the Playwright acceptance test against the Go‑served build with CSP active.

## 2026-07-14 — Phase 2: Android reporter

### D-0021 · Reporting runs in a background FlutterEngine hosted by the foreground service
- **Alternatives:** re‑implement crypto/queue/HTTP in Kotlin; foreground‑only (main‑isolate) reporting; a paid background‑location plugin.
- **Reason:** The native `LocationService` (foregroundServiceType=location) hosts its own `FlutterEngine` running the `aulLocationServiceMain` Dart entrypoint, and forwards each FusedLocation fix to it over a MethodChannel. This **reuses the tested Dart pipeline** (XChaCha20 sealing, drift offline queue, dio sender) instead of duplicating crypto in Kotlin, and keeps working with the UI gone or after reboot (`BootReceiver` restarts the service, which re‑creates the engine). Plugins are registered on the background engine so `flutter_secure_storage`/`path_provider` work headless; the isolate reads server URL, session, and per‑circle keys it needs from secure storage. Per spec, background location is native platform channels — not a third‑party plugin.

### D-0022 · Dart crypto uses libsodium for AEAD/X25519 and pure‑Dart SHA‑256 for the safety code
- **Alternatives:** libsodium for everything (incl. SHA‑256 via the sumo build); a pure‑Dart AEAD.
- **Reason:** Ping/place sealing (XChaCha20‑Poly1305), the identity keypair (X25519), and sealed key envelopes (`crypto_box_seal`) use the `sodium` package (real libsodium, bundled per‑ABI in the APK) — no hand‑rolled crypto. The **safety‑code fingerprint uses SHA‑256 via `package:crypto`** (pure Dart) because SHA‑256 is byte‑identical across Go/Dart/JS and avoids depending on the libsodium "sumo" build for one hash; it is validated against the committed vectors. Both are cross‑checked against the Go reference in `/vectors/crypto-vectors.json` (safety code + XChaCha20 encrypt/decrypt) by headless `flutter test`.

### D-0023 · APK build memory capped; CI builds a single ABI
- **Alternatives:** default 8 GiB Gradle heap; build all ABIs in CI.
- **Reason:** The default Flutter template sets `-Xmx8G`, which OOM‑kills the Gradle daemon on smaller build hosts. Capped to `-Xmx2560m` with `org.gradle.parallel=false`. CI builds `--target-platform android-arm64` (one ABI) for a fast build‑validation; release builds produce all ABIs. (The dev sandbox here has ~6.6 GiB RAM, which is why this surfaced.)

### D-0018 · Map tiles: OpenFreeMap (default), self‑host optional
- **Alternatives:** MapTiler / Mapbox (API keys, tracking, paid), Stadia, Google Maps SDK (banned by spec), raw OpenStreetMap raster.
- **Reason:** OpenFreeMap serves free vector tiles with **no API key and no request tracking**, is fully self‑hostable (Planetiler/OpenMapTiles), and renders through MapLibre — a perfect fit for a privacy product and for our fixed stack (no Google Maps SDK). The client points MapLibre at `https://tiles.openfreemap.org` by default; self‑hosters set `TILES_ORIGIN` (already wired into CSP and the compose `tiles` profile). Chosen by the maintainer.

### D-0019 · Phase‑2 pings are already end‑to‑end encrypted with the circle key from the invite fragment (no plaintext, no crypto stub)
- **Alternatives:** run the reporter in trusted (plaintext) mode until Phase 4; stub encryption.
- **Reason:** The circle key `K_c` distribution that Phase 2 needs is the *simple* half of the E2EE design and is available now: the owner generates `K_c` on device at circle creation, and joiners receive it in the invite‑link **URL fragment** (never sent to the server). So the reporter can seal pings with real **XChaCha20‑Poly1305** under `K_c` from day one — honoring "no stubs in crypto." The reporter also generates its **X25519 identity keypair** at registration (private key in `flutter_secure_storage`, public key sent to the server). What remains for **Phase 4** is the *hard* half: multi‑device key envelopes, key rotation, and the safety‑code verification UI. Cross‑language interop (Dart⇄Go XChaCha20 + SHA‑256 safety code) is pinned by the committed `/vectors/crypto-vectors.json`.

### D-0020 · Phase‑2 device validation strategy given no physical phone in CI
- **Alternatives:** claim the 24‑hour battery criterion as tested; skip runtime validation.
- **Reason:** The environment has an Android emulator (KVM available) but no physical device, so the literal "phone in pocket for a day, ≤3% battery" criterion cannot be measured honestly. Instead Phase 2 validates: (1) `flutter test` — Dart crypto against the cross‑language vectors, the offline queue, and the **adaptive scheduler** (profile→interval logic, the battery‑relevant behavior) as pure unit tests; (2) a **built APK**; (3) an **emulator smoke test** driving onboarding and posting a real (encrypted) ping to the live server (the "track lands on server" half of the criterion); (4) an in‑app **debug battery/accounting screen** for real‑device measurement by testers. The battery *target* is enforced by design (adaptive intervals, batching) and documented as pending on‑device confirmation.

## 2026-07-14 — Phase 1: server core

### D-0001 · Migrations tool: `goose` (SQL migrations), not an ORM or embedded custom runner
- **Alternatives:** `golang-migrate`, `atlas`, hand‑rolled `embed.FS` runner, ORM auto‑migrate.
- **Reason:** `goose` supports plain SQL up/down, is already available in the toolchain, embeds cleanly via `embed.FS`, and runs both as a CLI (dev) and programmatically (server startup). We keep migrations as reviewable SQL (important for a security product — no ORM magic touching auth/crypto columns). `sqlc` reads the same SQL for type‑safe queries.

### D-0002 · Type‑safe queries via `sqlc` generating `pgx/v5` code
- **Alternatives:** raw `pgx` with hand‑written scanning, `sqlx`, GORM/ent ORM.
- **Reason:** The spec mandates parameterized queries only. `sqlc` compiles `.sql` files into typed Go — every query is parameterized by construction, eliminating an entire class of SQL‑injection bugs, and there is zero runtime reflection. ORMs were rejected: they obscure the exact SQL, which is unacceptable when the queries touch password hashes and session tokens.

### D-0003 · Session tokens are opaque 256‑bit random strings, hashed at rest (SHA‑256), not JWTs
- **Alternatives:** JWT/PASETO stateless tokens.
- **Reason:** The spec requires revocation by `device_id` and refresh rotation. Stateless tokens cannot be revoked before expiry without a denylist (which reintroduces state). Opaque tokens are a single DB lookup, trivially revocable, and leak nothing if the DB read replica is compromised because only the SHA‑256 of the token is stored. Access TTL 15 min, refresh TTL 30 days, refresh rotates on every use (reuse of an old refresh token revokes the whole chain — theft detection).

### D-0004 · Password hashing: Argon2id, m=64 MiB, t=3, p=4, 16‑byte salt, 32‑byte output
- **Alternatives:** bcrypt, scrypt, PBKDF2.
- **Reason:** Mandated by spec and is current OWASP guidance. Parameters are stored in the encoded hash string (`$argon2id$v=19$m=65536,t=3,p=4$...`) so they can be tuned later and old hashes still verify. Implemented over `golang.org/x/crypto/argon2` (the reference Go implementation) — **not** self‑written.

### D-0005 · `pings` table partitioned by time using native PostgreSQL declarative range partitioning (monthly), with a maintenance job pre‑creating partitions
- **Alternatives:** single unpartitioned table + index; `pg_partman` extension; partition by circle.
- **Reason:** Pings are high‑volume, append‑mostly, and deleted by age (retention). Range partitioning by `server_ts` makes retention a `DROP`/`DETACH` of whole partitions (O(1), no bloat) instead of mass `DELETE`. Native partitioning avoids an extra extension dependency; a background job creates next‑month partitions ahead of time. Time (not circle) partitioning because retention is time‑based and cross‑circle.

### D-0006 · E2EE‑first schema from day one; "trusted server" (plaintext) is a config flag, default OFF
- **Alternatives:** ship plaintext first, retrofit E2EE in Phase 4.
- **Reason:** The columns (`nonce`, `ciphertext`) are E2EE‑shaped now even though Phase 1 clients send trusted‑mode payloads. Retrofitting encryption into a schema designed for plaintext is where E2EE products leak. `TRUSTED_SERVER_MODE` (default `false`) gates any server‑side plaintext handling and PostGIS geometry columns. **No plaintext coordinate is ever logged**, in either mode.

### D-0007 · One binary serves REST + WebSocket + embedded web assets + APK downloads
- **Alternatives:** separate services/containers per concern; nginx for static.
- **Reason:** Spec mandate ("Один бинарь"). `embed.FS` bundles the built web app and a manifest of downloadable APKs. Self‑hosters run one process. In dev, a `DEV_STATIC_DIR` env var serves from disk instead of the embedded FS so we don't rebuild Go to see web changes.

### D-0008 · Rate limiting: in‑process token bucket keyed by IP and by account, pluggable to Redis later
- **Alternatives:** Redis‑backed from day one; nginx/Caddy‑level limiting only.
- **Reason:** Phase 1 is single‑instance; an in‑process `golang.org/x/time/rate` limiter with a sharded, TTL‑evicted map is correct, dependency‑free, and testable. The limiter is behind a `Limiter` interface so a Redis implementation drops in for multi‑instance deploys without touching handlers. Edge (Caddy) limiting is complementary, not a substitute (it can't see accounts).

### D-0009 · WebSocket realtime via a single in‑process hub; fan‑out by circle; polling fallback
- **Alternatives:** Postgres `LISTEN/NOTIFY` fan‑out, external pub/sub (NATS/Redis).
- **Reason:** Single‑instance Phase 1 → a hub goroutine with per‑connection send channels and per‑circle subscription sets is the simplest correct design. Handlers publish an event to the hub after committing to the DB. For multi‑instance we will bridge hubs over Postgres `LISTEN/NOTIFY` (recorded as a future decision). Clients that can't hold a socket poll `GET .../pings` every 30 s.

### D-0010 · Config strictly via environment variables, validated at startup; fail fast
- **Alternatives:** config files (YAML/TOML), flags.
- **Reason:** Spec mandate + 12‑factor friendliness for Docker. A single `config.Load()` parses and validates all env, returns a typed struct, and refuses to boot on missing/invalid required secrets (e.g. `SESSION_HASH_PEPPER`, `DATABASE_URL`). No secret has a usable default.

### D-0016 · Fixes from the Phase-1 adversarial security review
An adversarial review (reviewers per dimension → independent verifier per finding) produced these confirmed fixes:
- **Real client IP resolution (HIGH).** `ClientIP` trusted the *leftmost* `X-Forwarded-For` hop, which a client controls (a proxy appends the real peer), letting an attacker rotate/forge IPs to bypass per‑IP rate limiting and lockout and to poison audit IPs. Now it prefers the proxy‑set `X-Real-IP` and, for XFF, uses only the *rightmost* (proxy‑appended) hop. (`internal/httpx/context.go`, regression tests added.)
- **Atomic refresh rotation (MEDIUM).** `RotateSession` was keyed on session id only, so two concurrent refreshes of one token could both succeed and evade reuse detection. It is now a compare‑and‑swap (`WHERE id=$1 AND refresh_token_hash=$old AND revoked_at IS NULL`); a 0‑row result is treated as reuse and revokes the session. (`db/queries/sessions.sql`, `internal/auth/service.go`, concurrent‑double‑spend test added.)
- **Instant‑leave feed cutoff (MEDIUM).** A removed/departed member's live WebSocket kept receiving the circle's events. The hub gained `EvictUser`/`EvictCircle`, invoked on member removal, leave, and circle deletion, so the feed stops immediately — enforcing the anti‑stalking instant‑leave guarantee. (`internal/realtime/hub.go`, `internal/httpapi/circles.go`, eviction test added.)
- **Login‑attempt metadata minimization (MEDIUM).** The source IP is now stored on `login_attempts` only when IP logging is enabled, and the table is pruned to 1 day (only the minutes‑long lockout window needs it), bounding the email↔IP mapping well below the IP‑log retention window. (`internal/auth/*`, `internal/retention/retention.go`.)
- **Registration enumeration (LOW).** Registration reveals whether an email exists (409). Accepted for v1; full mitigation needs email verification (Phase 7). Registration is IP rate‑limited (now non‑spoofable). Tracked in TODO/THREAT_MODEL.

### D-0017 · Fixes from the Phase-1 security review round 2 (DoS hardening)
A second adversarial pass focused on access-control/IDOR and input/DoS. **No authorization/IDOR defects were found** (membership/ownership enforcement held). Three DoS-hardening findings were confirmed and fixed:
- **Key-envelope write amplification (MED).** `POST /v1/key-envelopes` had no rate limit, allowed 1000 rows/request, and inserted duplicates that retention never pruned. Now: per-user rate limit (30/min), cap lowered to 200, an upsert keyed on a new unique index `(circle_id, recipient_device_id, key_epoch)` so a recipient holds exactly one sealed key per epoch (bounded storage), and a retention step prunes consumed (>7d) and stale unconsumed (>30d) envelopes. (`migrations/00004`, `db/queries/key_envelopes.sql`, `internal/httpapi/keyenvelopes.go`, `internal/retention`.)
- **Unbounded WebSocket connections (MED).** `/v1/realtime` had no upgrade rate limit and no connection cap. Now: a per-IP upgrade limiter (30/min) and hub admission control with per-user (10) and global (10000) ceilings enforced *before* the upgrade (`Admit`/`Release`), returning 429/503. (`internal/realtime/hub.go`, `internal/httpapi/ws.go`, cap test added.)
- **Unbounded pending-envelope read (LOW).** `PendingEnvelopesForDevice` had no `LIMIT`; added `LIMIT 500` (and the upsert above bounds the queue anyway).
- Dismissed by the verifier: Argon2 CPU DoS behind the per-IP auth limiter (the per-IP cap — now non-spoofable per D-0016 — bounds it; the cost is intentional).

### D-0014 · Go module floor set to 1.25 (within the spec's "Go 1.24+")
- **Alternatives:** pin dependencies to their last Go‑1.24‑compatible releases to keep a 1.24 floor.
- **Reason:** Current `pgx/v5`, `goose/v3`, and the `golang.org/x/*` libraries now declare a `go 1.25` directive, which transitively raises this module's floor. Pinning back to older releases means shipping a security product on less‑maintained dependency versions — the wrong trade. Go 1.25 satisfies the spec's "Go 1.24+" (a *minimum*), so the floor is 1.25, the Docker build image is `golang:1.25-alpine`, and CI uses Go 1.25. The stack choice (Go) is unchanged.

### D-0015 · Integration tests run serially (`go test -p 1`) against one shared database
- **Alternatives:** a fresh database/schema per test package; testcontainers per package.
- **Reason:** Integration tests truncate all tables for isolation. Go runs package test binaries in parallel by default, so concurrent packages would truncate each other's in‑flight rows (observed as spurious 401s and count mismatches). `-p 1` serializes packages, which is simple and deterministic; per‑package databases were rejected as unnecessary complexity for Phase 1. Documented in CONTRIBUTING and enforced in CI.

### D-0012 · Ping timestamp is split into `captured_at` (authoritative, partition + dedup key) and `received_at` (server clock, audit)
- **Alternatives:** single `server_ts = now()` column; a separate unpartitioned `ping_receipts` idempotency table; app-level windowed dedup.
- **Reason:** Idempotent batch inserts (`ON CONFLICT DO NOTHING`) need a unique index, and PostgreSQL requires the partition key inside every unique index on a partitioned table. A server‑assigned `now()` timestamp changes on retry, so it can't anchor dedup. Instead the ping carries a **client‑declared `captured_at`** (stable across retries) which is the partition key, retention key, and part of the dedup index `(device_id, client_id, captured_at)`. The server validates `captured_at ∈ [now − MaxRetentionDays, now + 5m]` to bound partition placement and reject junk; a separate server‑clock `received_at` is kept for audit. This maps the spec's `server_ts` to `captured_at` while making exactly‑once ingestion race‑safe with no companion table and O(1) partition‑drop retention. A companion receipts table was rejected because it has the same row cardinality as pings and reintroduces the mass‑DELETE cost partitioning exists to avoid.

### D-0013 · Refresh‑token reuse detection via current + previous hash on the session row
- **Alternatives:** full token‑family history table; no reuse detection; JWT with jti denylist.
- **Reason:** Each session stores the current refresh hash and the immediately previous one. Normal rotation matches the current hash and shifts it to previous. Presenting the *previous* (already‑rotated) hash means the token was replayed — a theft signal — and revokes the whole session. This catches the standard stolen‑refresh scenario without an unbounded history table, and is documented as a Phase‑1 heuristic (full family trees can come later).

### D-0011 · Server crypto layer is real and tested even in Phase 1
- **Alternatives:** stub crypto until Phase 4.
- **Reason:** Explicit non‑negotiable from the brief: no stubs in crypto/security. Phase 1 ships and tests: Argon2id hashing/verification, constant‑time token hashing, `crypto/rand` token generation, and the safety‑code (emoji fingerprint) derivation that clients will mirror — with cross‑language test vectors checked in so Dart/JS implementations can be verified against Go in later phases.
